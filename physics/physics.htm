AGE Physics Library

 

 

1.      Conventions

In this document, the following conventions are used:

·         Bold face is used for the names of code libraries (i.e., .dsp files). Note that library names always start with the prefix “ph”.

·         Italics are used for code modules.

·         The AGE Physics Library as a whole, including all six libraries detailed below, is referred to as the “AGE Physics Library”, or just as “the physics library”. This should not be confused with the physics library, which is a code library, and will be referenced in bold face.

 

2.      Main Concepts

The following are the main concepts with which you must be familiar in order to understand and effectively use the physics library:

·         Bound

A “bound” is a geometric representation of an object, whether defined mathematically (e.g., a sphere, which requires only a radius to define its topology) with a collection of polygons, or by some other method. With few exceptions, a bound is an archetype which does not possess a location or rotation. Therefore, many objects can reference the same bound, allowing you to create a large number of instances with minimal local data (such as matrices, damage states, etc.).

·         ICS

The “phInertialCS”, or “ICS”, is a structure which contains information on the location, velocity, angular velocity, mass, density, moments of inertia, torque, etc., of an object. In short, everything with which to determine where an object is, and where it will be. In many ways the ICS is a complement to the bound – it knows nothing of the archetypal collideable geometry of an object type, but it knows where an object instance is and where it’s going. After a collision it determines the resulting forces, and every frame it updates position and rotation.

·         Collider

A “collider” is an object which serves as a container for a bound and an ICS. It is, in essence, a physically active object with a collideable geometry, position, rotation, velocities, etc. – everything necessary for full simulation of physical motion.

·         Instance

An “instance” is a collideable object which doesn’t have a collider, but does have a bound, a position, and a rotation (possibly only in Y, depending on your application). It’s waiting for something to hit it so that it can be assigned a collider, waken, and become fully physically active. The goal is to have thousands of instances (each of which is very cheap) and very few colliders (which are comparatively expensive), giving the world complexity without making it computationally prohibitive.

 

3.      Libraries

 

 


There are four core physics libraries in AGE, phcore, phinertia, phbound, and phcollide. Additionally, there is one library, physics, which provides several modules which can be useful as sample code and when creating basic applications, but are designed from the standpoint of simplicity. Finally, there is a library named phtest which contains testing modules. The above graph shows the dependencies of the various libraries – note that phinertia and phbound don’t rely on each other. Note that except in extreme situations, you should consider the four core libraries as read-only, whereas you should make a copy of physics, use its code as a starting point, and alter as necessary.

 

·         phcore

This library contains the most essential elements for the whole of AGE Physics, and is required by all other physics libraries. phcore contains the following modules:

-          constants

Contains important constants for physics. At present, these consist only of the gravity constant, several penetration constants, and some debug defines.

-          convexpoly

A type of polygon which knows which only has one side.

-          edge

A polygon edge.

-          isect

An intersection. This type is heavily used during collision detection, and is derived from a class found in isectpoint.

-          isectpoint

The most basic information used in storing an intersection.

-          line

A set of courtesy functions that allow the buffered display of lines on screen for debugging.

-          material

Physics materials.

-          node

Basic linked-list functionality (it was lacking elsewhere in AGE).

-          phmath

Contains more complex math routines only used (currently) in AGE (e.g., solutions to a quartic).

-          poly

A polygon. Note that in VTune, the segment/polygon collision routines in this module are almost always the prime villain in framerate suckitude™. And, although there will no doubt be a lot of effort in hand-tuning these in assembly, avoiding these routines is by far the best way of improving physics performance. Note that it takes almost the same amount of time to test a quad as it does to test a triangle, so the more you can optimize your bounds to use quads instead of tri’s, the better off you will be. Please refer to the optbound tool in phtest.

-          segment

A line segment.

-          treenode

Basic tree functionality (currently incomplete, but planned for use with hierarchically connected bounds).

 

·         phinertia

This library contains all functionality related to ICS’s, and the location and movement of objects. It requires the phcore library, and contains the following modules:

-          constraint

Adds constraints to the relative motions of two ICS’s.

-          inertia

The ICS (actually, ICS, or “inertial coordinate system”, is a horrible misnomer. Due to historical pressures, however, we continue to use this terminology – deal).

-          joint

General code for handling 1, 2, and 3 degree of freedom (DOF) joints.

-          spring

Code for simulating springs.

 

·         phbound

This library contains all functionality related to bounds and most of the collision code. It requires the phcore library, and contains the following modules:

-          bound

The base class from which all other bounds are derived. In the event you wish to create your own custom bound, you should derive it from this class.

-          boundbox

A bound type that represents a box, defined as a rectilinear, axis-aligned six-sided polyhedron. This type is extremely useful, as it is both common as well as much faster than the identical shape represented by a polygonal geometry in boundgeom (see below).

-          boundcomposite

A bound made up of multiple sub-bounds. This type is still a work in progress.

-          boundcontainer

A container for a boundcomposite. In many cases a boundcomposite will have internal moving parts (e.g., a biped), in which case you will want to set the internal matrices appropriately before doing collision detection – this module provides a method through which you can perform whatever initialization is necessary before the actual collision testing.

-          boundcylinder

A perfect cylinder. Note that this module is currently only about 80% done, and may very well not be completed due to its inherent slow performance (due to the flat ends of the cylinder, it’s necessary to solve a quartic, and possibly an 8th-degree polynomial in certain cases). At the present time, it’s suggested that you use either boundhotdogs or boundgeoms to represent your cylinders (see below).

-          boundforce

The base class for bounds which impart forces. Examples would be explosions, fluid, etc. Note that these are only kind of bounds, but have the same basic set of functionality, so are derived from the same base class. They also break the rules by having positions and rotations.

-          boundgeom

A geometric bound formed of polygons. This class contains loading routines as well as several optimizing preprocessing routines to get rid of degenerate polygons, transform triangles into quads, etc.

-          boundhotdog

A cylinder with hemispheres on the ends. This type is second only to the sphere in speed, and is recommended for use whenever possible.

-          boundpolygonal

A pure class from which all polygonally-based bounds are derived. The two main examples are boundbox and boundgeom.

-          boundsphere

A perfect sphere. This is the fastest type to collide against.

-          boundterrain

A boundgeom that is divided into a uniform grid of polygon “buckets”. All polygons which intersect a cell in the grid will be stored in that bucket, and when a bound (or line segment) is tested against the boundterrain, it will only be tested against those buckets it intersects.

-          collision

Contains shared collision routines.

-          collisionprim

Contains basic collision primitives (e.g., line vs. sphere, sphere vs. poly, etc.).

-          forcesphere

A perfect sphere which imparts an outward force to any object it intersects. Useful for explosions. Derived from boundforce.

-          generator

Code used in generating boundterrain buckets.

-          impact

A class which contains basic information on two bounds which have collided.

-          metaboundgeom

A class which references a boundgeom but is able to store its own copy of vertices. Currently used only in R&D for real-time deformation of objects.

 

·         phcollide

This library combines the bound and ICS functionality for full physical simulation. It relies on the phcore, phinertia, and phbound libraries, and contains the following modules:

-          collider

This is the main module of the library. It defines the phCollider class, which serves as a container for a bound, an ICS, and a sleep (see below). One collider represents and controls a single physically active object.

-          collision

Contains shared collision routines.

-          contact

A class which maintains information on a single point of contact between two objects. Contacts are important for two reasons – first, they improve the speed of collision detection, as you already know where two objects are touching, and second, they allow objects to come to rest against each other, allowing stacking.

-          contactmgr

A class which manages the contacts.

-          impact

Derived from the impact class defined in phbound, this class provides extra functionality related to colliders. Note that this class doesn’t have any extra data members, so it is possible to use it interchangeably with its superclass.

-          instance

Defines the most basic form of instance. It has a type ID, a matrix, and a bit vector for status information. Note that the type ID can be used by your program in any way you see fit – in the default managers defined in physics, it is an index into an array of bound information, although this could just as easily be a pointer to a bound, or to a struct with game-specific information (presumably also including a pointer to a bound). It is important for this struct to be as small as possible so that a great number of them may be stored.

-          sleep

A class which controls how fast and when an object will fall asleep.

-          sleepmgr

Currently obsolete, but kept around just in case.

 

·         physics

This library contains a set of general-purpose routines which may or may not be of use to your project. They are provided as examples, but it is expected that you will change them as necessary to more accurately reflect the needs of your application. It depends on phcore, phinertia, phbound, and phcollide.

-          boundmgr

A singleton which manages bounds and their requisite information. It is likely that you will be able to use the boundmgr as is – it provides a simple interface to load and manage bounds, including optimizations for initialization of colliders at promotion time, and the ability to store metamodels for bounds. You may want to modify it to initialize special types used by your application.

-          collidermgr

Manages all instances, colliders, ICS’s, and sleeps. Performs collisions every frame, promotes and demotes objects appropriately, renders models based on object locations. It is likely that you will want to use this as a starting point for your application’s collision management, but it is probably only appropriate for use as is in demos or smaller projects. Its main disadvantage is that it doesn’t have any form of room management.

-          forcecloth

A type of bound being used in R&D.

-          forceliquid

A type of bound being used in R&D.

-          instmgr

A list manager for instances. Planned for use in the room manager.

-          object

A very simple class which represents a single object in the collidermgr. It maintains information on its own instance index, and its initial matrix. When it’s reset, it re-initializes its initial matrix. Currently only used in some demos.

-          room

Still in the initial stages of development. A class which represents a physics room, i.e., a localized region within which objects can interact.

-          roommgr

Still in the initial stages of development. This class will eventually go a long way to replacing the collidermgr – it will maintain a list of rooms, and will only test an object against objects in the rooms it intersects. This will allow the creation and maintenance of a much larger database of objects, as active objects will only be tested against local objects, not those in distant rooms.

 

·         phtest

The modules of phtest are used for testing purposes or as tools.

-          block

An old piece of code with much the same functionality of object in physics. Its only current use (to my knowledge) is in launchingball.

-          launchingball

An object which can be launched (useful in stacking demos).

-          maketerrain

A tool which processes a boundterrain and saves out its data. Boundterrains generally take a long time to load because there’s a lot of preprocessing to be done, so use of this tool in the asset pipeline speeds load time tremendously (when you have large boundterrains, that is).

-          optbound

A tool which optimizes bound (bnd) files by removing duplicate vertices, edges, and polygons, and by converting coplanar, attached triangles into quads (with appropriate checks for convexity). This has a huge impact on performance.

-          phage

Generic tester I use for testing various parts of the physics library.

-          phroom

Tester for rooms.

-          phtable

Tester originally created for testing boundcomposites, but now just a general tester.

-          roomtest

Tester for rooms.

-          roomtool

Tool for creating simple room files. Currently incomplete.

 

4.      Customization

Although there’s a lot of base functionality built into the physics library, it’s likely that you’ll have to customize the following areas:

·         collidermgr

The collider manager, as it stands, is really only useful for a world which contains several hundred objects or fewer. The reason for this is that it has no way of sectioning the world into multiple areas (i.e., “rooms”), which means that every frame it has to do sphere tests between every active collider and every instance. Ideally, you would want the time to be a function of the number of active colliders, with very little relation to the number of sleeping instances in the world. And, the needs of each project will necessitate different approaches, which makes this the biggest and most important area of customization for a project.

·         New Bounds

It’s also likely that your project will have a bound type unique to it. An example of this is the SDL in Midnight Club, or the offroad level bound type in Smuggler’s Run. The most important piece of functionality required by a new bound is the various TestSegment() functions, each of which tests a line segment against the bound. These are:

-        TestEdge()

Returns the closest and farthest intersections between the segment and the bound. This is used in collision detection with polygonal objects.

-        TestProbe()

Returns the closest intersection between the segment and the bound. Also used in collision detection with polygonal objects, although it has many other uses (e.g., wheel probes, gunshots, etc.).

-        TestAI()

Returns the first intersection discovered between the segment and the bound. This is not used internally by the physics code, and is only important if your application has a need for it (it was originally conceived as being useful for pathfinding, although that usefulness is questionable).

Depending on your application, you will also probably have to override the TestSphere() function, which determines whether a sphere has collided with your bound, and if so, returns the position, normal, and depth of impact. Finally, you will have to write custom routines for collision detection between your object and any non-polygonal object with which it’s important for your bound to collide. Remember, although you may think that the hotdog is a pretty exotic shape, it’s fast, and may be just the thing for lamps, trees, parking meters, biped limbs, etc., which means that you’re probably going to need a TestMyBoundVsHotdog() routine somewhere along the way. One word of caution – a common mistake is to forget to fill in certain pieces of information in the intersection record, but this will cause bad behavior. There’s lots of sample code lying around in the other bounds, and there are people to ask – when in doubt, fill in more info.

 

5.      Getting Started

Ok, now that you’ve had an overview of the pieces, I’ll go over a simple piece of code that initializes AGE, sets up the physics manager, creates an box and a flat plane, and drops the box when you hit the space bar.

 

int Main()

{

   ageInit("//zen/projects/age/assets/physics");

   ageBeginGfx();

 

   devPolarCam PC;

   PC.Init(10,2.0f,0.3f);

 

   RSTATE.SetLighting(true);

   RSTATE.SetAmbient(mkrgb(30,30,30));

 

   gfxLight light;

 

   light.type = lightDirectional;

   light.diffuse.Set(1,1,1,1);

   light.ambient.Set(0,0,0,1);

   light.dir.Set(-1.0,-0.5f,0.5f);

   light.range = 100.0f;

   RSTATE.SetLight(0, &light);

   RSTATE.LightEnable(0, true);

 

The above code is pure AGE setup, with no relation to the physics library. Now comes the physics setup.

 

   /////////////////////////////////////////////////

   // BEGIN PROGRAM-SPECIFIC SECTION OF CODE

   phColliderMgr *collidermgr = new phColliderMgr();

   phColliderMgr::SetActiveInstance(collidermgr);

   COLLIDERMGR->Init(200,1000,100);

 

The above section instantiates a collidermgr and sets it as the active collidermgr. You can have multiple collidermgr’s, but when one is set as “active”, it acts as a singleton. The Init() function specifies how many colliders, instances, and broken pieces may exist (broken piece functionality is currently, er, broken).

 

   /////////////////////

   // ground

   int index;

   char terrain[] = "big_plane";

   BOUNDMGR.GetArchetype(phBound::GEOMETRY,terrain,Vector3(1,1,1),1,NULL,

&index);

   BOUNDMGR.SetModel(index,terrain);

   COLLIDERMGR->RequestCollider(index,NULL,phColliderMgr::FIXED);

 

Get a bound from the bound manager. If the bound has not yet been loaded (as is the case here) the boundmgr will automatically load it. Normally, when you’re instantiating a bound you won’t request a collider for it, but in this case you want to because this bound is going to act as terrain. All “fixed” objects have to be set up specially, as they have neither ICS’s nor sleeps. Below is an example of a normal physical object being instantiated.

 

   /////////////////////

   // object

   int instance;

   Matrix34 initial;

 

   BOUNDMGR.GetArchetype(phBound::BOX,"cube",Vector3(1,1,1),1,NULL,&index);

   BOUNDMGR.SetModel(index,"cube");

 

   initial.MakeRotateX(1);

   initial.d.Set(0,3,0);

   instance = COLLIDERMGR->CreateInstance(index,initial);

 

Here, we get the index for a bound (which also loads or creates the bound if it hasn’t already been loaded or created), set up its initial matrix, and create an instance of the bound with the specified matrix. We also set a model for the object so that it will be automatically rendered by the collidermgr. Your application will probably want to do its own rendering of objects, but in this simple case it’s easier to just let the collidermgr take care of it. Note that the instance does not currently have a collider – it is “demoted”, or asleep, and will stay that way until another object hits it, or until it is explicitly wakened.

 

   // END PROGRAM-SPECIFIC SECTION OF CODE

   /////////////////////////////////////////////////

 

   bool Pause = false;

 

Pause is used below so that you can single-step through a physics routine for debugging purposes. Basically, if you don’t Update() the collidermgr, no physics happens. See below for details.

 

   do

   {

          ageBeginFrame();

 

          PC.Update();

          PIPE.SetCamera(PC.GetWorldMtx());

 

          /////////////////////////////////////////////////

          // BEGIN PROGRAM-SPECIFIC SECTION OF CODE

 

          if (ioKeyboard::KeyPressed(KEY_SPACE))

          {

                 if (COLLIDERMGR->GetInstance(instance).IsDemoted())

                        COLLIDERMGR->PromoteInstance(instance);

          }

 

If we hit the space bar, promote the instance (if necessary).

 

          if (ioKeyboard::KeyPressed(KEY_F7))

          {

                 COLLIDERMGR->DemoteInstance(instance);

                 COLLIDERMGR->GetInstance(instance).Matrix = initial;

                 COLLIDERMGR->Reset();

          }

 

If we hit the F7 key, reset the scene to its original state.

 

          if (Pause == false || KEYBOARD.KeyPressed(KEY_F8))

                 COLLIDERMGR->Update();

          Pause = !KEYBOARD.KeyPressed(KEY_SPACE) &&

(Pause || KEYBOARD.KeyPressed(KEY_F8));

 

If we hit the F8 key, pause the scene, or single step through the scene. If we hit the space bar, continue normally.

 

          COLLIDERMGR->Draw();

 

Render all objects managed by the collidermgr.

 

          // END PROGRAM-SPECIFIC SECTION OF CODE

          /////////////////////////////////////////////////

 

          ageEndFrame();

   } while (ioKeyboard::KeyPressed(KEY_ESCAPE) == false);

 

   delete collidermgr;

 

   ageEndGfx();

 

   return 0;

}
